# ADR-004: 兩階段驗證（TOTP）

- 日期：2026-06-02
- 狀態：Accepted
- 決策者：AI Agent + 待 BE Lead 確認

## Context

Spec AC-2FA-1 / AC-2FA-2 / US-3 要求支援 2FA。需決定：
- TOTP / SMS / WebAuthn 哪種
- enrollment 流程是否需要 pending state
- 登入第二階段如何銜接（state 怎麼保存）

## Options

### A：TOTP（RFC 6238）
- 優：離線即可、無 SMS 成本、實作標準化
- 缺：使用者需安裝 authenticator app

### B：SMS OTP
- 優：使用者門檻低
- 缺：成本（Twilio）、SIM swap 攻擊風險

### C：WebAuthn / Passkey
- 優：UX 與安全性都好
- 缺：實作較複雜、設備支援不齊

## Decision

**選 Option A（TOTP）。** 用 `otplib` 套件。

### enrollment 設計
- 直接寫入 `user.totpSecret` → 立即啟用
- 不引入 pending state（YAGNI；schema 也未預留）
- 若 user 失去 authenticator → 可重新 enroll 覆寫舊 secret（合理 trade-off）

### 第二階段 challenge_id 設計
- HMAC-簽 `2fa.<nonce>.<userId>.<exp>.<sig>`，5 分鐘 TTL
- 與 csrfState 同樣 stateless（不存 DB）
- 改用 `domain prefix` (`2fa.`) 防 token 混用（不同用途 secret 不可互換使用）

### 驗證
- otplib `authenticator.verify` 內部使用 timing-safe 比較
- 額外加 6 位數字格式檢查防雜訊

## Consequences

- 正面：實作簡單、無第三方依賴、stateless 設計可水平擴展
- 負面：
  - 沒有 pending state，使用者點下 enroll 後即生效，UX 上需明確提示「請完成驗證否則下次無法登入」
  - 沒有 backup codes（下個 sprint）
- 待辦：
  - PR #5 後加 backup codes
  - 後續考慮 WebAuthn 作為更強選項
